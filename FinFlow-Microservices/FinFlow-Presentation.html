Context
Problem Statement
Why build FinFlow?

Challenges

  • Loan applications handled manually — slow and error-prone
  • No real-time tracking of application status
  • Document verification is disconnected from application flow
  • Monolithic architecture limits scalability
  • No audit trail for admin decisions

Solution

  • Microservices architecture for independent scaling
  • Event-driven communication via RabbitMQ
  • Real-time status updates across services
  • Centralized gateway with JWT security
  • Complete observability with Zipkin + logging

Security
Authentication Flow
JWT-based stateless authentication across microservices
1

Register — POST /auth/signup with name, email, password

2

Login — POST /auth/login → returns JWT token + role

3

Request — Client sends token in Authorization header

4

Gateway — Validates JWT, extracts userId and role

5

Forward — Injects X-User-Id + X-User-Role headers

6

Service — Reads headers, no JWT parsing needed

JWT Token Claims

{ "sub": "deepak@capg.com", "userId": 1, "role": "ROLE_APPLICANT", "iat": 1711670400, "exp": 1711756800 }

How It Works

Trace ID generated at API Gateway for each request

2

Propagated to every downstream service automatically

3

Span IDs created for each service hop

4

Collected by Zipkin at 100% sampling rate
